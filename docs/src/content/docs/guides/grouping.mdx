## What is Grouping

Grouping turns each row of your data into a point with up to four dimensions â€” **Name**, **XAxis**, **YAxis**, and **ZAxis** â€” so vizb can chart it meaningfully instead of as a flat list.

It always works in **two stages**:

1. **Build a label** for the row. For **CSV/JSON**, you pick the columns with `--group` and vizb joins their values with `/`. For **benchmarks**, the benchmark name *is* the label already (e.g. `BenchmarkSort/1024/QuickSort`).
2. **Split the label** into dimensions with `--group-pattern` (separator-based) or `--group-regex` (named captures). This stage is identical for both inputs.

The two flows differ only in stage 1 â€” where the label comes from â€” so the same pattern and regex syntax serves both.

## Why Grouping

Grouping creates multi-dimensional charts. Without it, every row is an isolated bar or point. With it, vizb can:

- **Separate charts** by logical category (e.g. one chart per algorithm or region)
- **Populate axes** with meaningful values (e.g. input size on the X-axis)
- **Create series** for variants (e.g. different data distributions on the Y-axis)

| Without Grouping | With Grouping |
|---|---|
| Single flat chart with all rows | Multiple charts grouped by name |
| No axis structure | X-axis and Y-axis with meaningful categories |
| Hard to compare across dimensions | Easy side-by-side comparison |

## The Dimensions

Grouping extracts up to four dimensions from each label:

| Dimension | Role | Description |
|---|---|---|
| **Name** (`name` / `n`) | Groups rows into separate charts | Rows sharing the same name appear in the same chart |
| **XAxis** (`xAxis` / `x`) | Sets the X-axis categories | Each unique value becomes a tick on the X-axis |
| **YAxis** (`yAxis` / `y`) | Creates series / variants | Each unique value becomes a separate series or bar group |
| **ZAxis** (`zAxis` / `z`) | Adds a depth axis â†’ 3D chart | Each unique value becomes a stacked depth layer in a [3D chart](/ui/3d-charts) |

<Aside type="note">
  At least one axis (`x` or `y`) is required for grouping to take effect. The `name` dimension is optional â€” when omitted, all rows go into a single chart. The `z` dimension requires **both** `x` and `y` to be present.
</Aside>

## Grouping CSV/JSON data

CSV and JSON rows have no benchmark name, so you tell vizb which columns carry the structure. The **`--group` / `-g`** flag selects them; their row values are joined with `/` into a label, which is then split by `--group-pattern` (or `--group-regex`) just like a benchmark name.

Take this `sales.csv`:

| region | product | sales | latency |
|---|---|---|---|
| USA | Widget | 100 | 12 |
| USA | Gadget | 80 | 15 |
| EU | Widget | 60 | 11 |

```bash
vizb sales.csv --group region,product -p y/x -o sales.html
```

For the first row, vizb joins the selected columns into the label `USA/Widget`, then the pattern `y/x` splits it:

| Selected columns | Label | Pattern `y/x` | Result |
|---|---|---|---|
| `region`, `product` | `USA/Widget` | `y` â†’ `USA`, `x` â†’ `Widget` | YAxis = USA, XAxis = Widget |

The columns you **don't** select decide the chart values:

- **Unselected numeric columns become series.** Here `sales` and `latency` each become a series (one `Stat` per column).
- **Non-numeric unselected columns are ignored.** A `notes` text column would simply be dropped.
- **No `--group` at all** â†’ dimensions stay empty, no rows are merged, and each row becomes one point carrying every numeric column as a series. Useful for a quick chart of a wide table.

<Aside type="note">
  Each `--group` column must match a header exactly. A missing column is fatal: `group column 'X' not found; available: [...]`.
</Aside>

### Aggregation

When `--group` is set, multiple rows can share the same `(name, x, y, z)` key. Vizb **sums** their values into a single point so a 200k-row export collapses into a few thousand chart points. You'll see it in the CLI output:

```text
ðŸ§® Aggregating 50000 rows...
âœ… Aggregated into 1200 grouped data points
```

<Aside type="caution">
  Aggregation is **summing**, and applies to **CSV/JSON only**. Benchmark parsers are never aggregated â€” repeated runs share a key but the UI averages them instead of summing.
</Aside>

See [Tabular Data (CSV & JSON)](/guides/data) for the full parser rules.

## Grouping benchmarks

Benchmark names already encode structure. A name like `BenchmarkSort/1024/QuickSort` carries the function, the input size, and the variant. There are no columns to pick, so `--group-pattern` / `--group-regex` do all the extraction, and `--group` only supplies human-readable axis **labels** (see [Axis labels](#axis-labels-with---group---g)).

```bash
go test -bench . | vizb -p n/x/y -o output.html
```

| Benchmark Name | Name | XAxis | YAxis |
|---|---|---|---|
| `BenchmarkSort/1024/QuickSort` | Sort | 1024 | QuickSort |
| `BenchmarkSort/2048/MergeSort` | Sort | 2048 | MergeSort |

Every metric the benchmark reports (time, memory, allocations, â€¦) becomes a series automatically â€” you only group the name.

## Pattern Syntax

The `--group-pattern` flag splits the label using a separator (`/` or `_`) and assigns each segment to a dimension. It applies to both flows â€” the benchmark name or the CSV/JSON label built from `--group`.

### Separators

- `/` â€” slash separator (the Go benchmark convention)
- `_` â€” underscore separator

### Components

| Component | Shorthand | Meaning |
|---|---|---|
| `name` | `n` | Chart group name |
| `xAxis` | `x` | X-axis values |
| `yAxis` | `y` | Y-axis series |
| `zAxis` | `z` | Z-axis depth layer (3D, requires `x` + `y`) |

The default pattern is `x` â€” a single segment mapped to the X-axis.

### Examples

**Slash-separated `n/x/y`:**

```bash
vizb bench.txt --group-pattern "n/x/y" -o output.html
```

| Label | Name | XAxis | YAxis |
|---|---|---|---|
| `BenchmarkSort/1024/QuickSort` | Sort | 1024 | QuickSort |
| `BenchmarkSort/2048/MergeSort` | Sort | 2048 | MergeSort |
| `BenchmarkSort/4096/QuickSort` | Sort | 4096 | QuickSort |

**Underscore-separated `n_y_x`:**

```bash
vizb bench.txt --group-pattern "n_y_x" -o output.html
```

| Label | Name | YAxis | XAxis |
|---|---|---|---|
| `BenchmarkHash_SHA256_1KB` | Hash | SHA256 | 1KB |
| `BenchmarkHash_MD5_4KB` | Hash | MD5 | 4KB |

**2D with `n/y` (no X-axis grouping):**

```bash
vizb bench.txt --group-pattern "n/y" -o output.html
```

| Label | Name | YAxis |
|---|---|---|
| `BenchmarkJSON/Marshal` | JSON | Marshal |
| `BenchmarkJSON/Unmarshal` | JSON | Unmarshal |

**Prefix skip with `/n/y` (leading slash discards the first segment):**

```bash
vizb bench.txt --group-pattern "/n/y" -o output.html
```

| Label | Name | YAxis |
|---|---|---|
| `BenchmarkTest/JSON/Marshal` | JSON | Marshal |
| `BenchmarkTest/XML/Parse` | XML | Parse |

**4D with `n/x/y/z` (renders a 3D chart):**

```bash
vizb bench.txt --group-pattern "n/x/y/z" -o output.html
```

| Label | Name | XAxis | YAxis | ZAxis |
|---|---|---|---|---|
| `BenchmarkSort/1024/QuickSort/warm` | Sort | 1024 | QuickSort | warm |
| `BenchmarkSort/2048/MergeSort/cold` | Sort | 2048 | MergeSort | cold |

<Aside type="note">
  A pattern must contain `x` or `y` (`pattern must contain xAxis (x) or yAxis (y)`). A `z` segment requires both `x` and `y` â€” vizb rejects `z` otherwise with `zAxis (z) requires both xAxis (x) and yAxis (y)`. See [3D Charts](/ui/3d-charts).
</Aside>

## Regex Syntax

The `--group-regex` flag uses Go-style named capture groups for full flexibility. It also runs on the label â€” a benchmark name, or the CSV/JSON label built from `--group`. Use it when the label does not follow a simple separator pattern. When set, regex takes precedence over `--group-pattern`.

### Named Capture Groups

| Group Name | Shorthand | Dimension |
|---|---|---|
| `(?<name>...)` | `(?<n>...)` | Chart group name |
| `(?<xAxis>...)` | `(?<x>...)` | X-axis values |
| `(?<yAxis>...)` | `(?<y>...)` | Y-axis series |
| `(?<zAxis>...)` | `(?<z>...)` | Z-axis depth layer (3D, requires `x` + `y`) |

### Examples

| Label | Regex | Extracted |
|---|---|---|
| `BenchmarkHashing64MD5` | `Hashing64(?<x>.*)` | X: MD5 |
| `BenchmarkJSONByMarshal` | `(?<x>.*)By(?<y>.*)` | X: JSON, Y: Marshal |
| `BenchmarkDecode/text=digits/level=speed` | `(?<n>.*)/text=(?<x>.*)/level=(?<y>.*)` | N: Decode, X: digits, Y: speed |

```bash
vizb bench.txt --group-regex "(?<n>.*)/text=(?<x>.*)/level=(?<y>.*)" -o output.html
```

<Aside type="note">
  The regex must capture `x` or `y` (`regex '...' does not contain x (xAxis) or y (yAxis)`), and a captured `z` still requires both `x` and `y`.
</Aside>

## Axis labels with `--group` / `-g`

`--group` carries human-readable labels for each dimension so charts render meaningful axis titles instead of raw identifiers. Labels map **positionally** to the `--group-pattern` parts, in order:

```bash
# Pattern n/y/x â†’ --group labels: Sort=name, algorithm=y, size=x
go test -bench . | vizb -p n/y/x --group Sort,algorithm,size -o output.html
```

| Pattern slot | `--group` entry | Axis label |
|---|---|---|
| `n` (name) | `Sort` | chart group label |
| `y` | `algorithm` | Y-axis title |
| `x` | `size` | X-axis title |

The same flag plays a different role depending on the parser:

- **Benchmark parsers** (`go`, `rs:*`, `js:*`): `--group` is **labels only** â€” there are no columns. It names the dimensions that `--group-pattern` already extracted from the benchmark name.
- **CSV/JSON parsers**: `--group` entries are **column/field names**. They select which columns build the label *and* serve as that dimension's label â€” one flag, both jobs.

<Aside type="note">
  Labels are included in the JSON output under `axisLabels` and displayed on the chart axes. The `name` label is preserved for the UI axis-swap feature, which lets you rotate dimensions while carrying their labels.
</Aside>

## Choosing a Method

<Tabs syncKey="method">
  <TabItem label="Pattern" icon="list">
    Use `--group-pattern` when the label follows a consistent separator format (`/` or `_`).

    ```bash
    vizb bench.txt --group-pattern "n/x/y" -o output.html
    ```

    Best for standard Go benchmarks (`BenchmarkFunc/size/variant`) and for CSV/JSON where the selected columns slot cleanly into dimensions.
  </TabItem>
  <TabItem label="Regex" icon="search">
    Use `--group-regex` when the label has a non-standard format or you need fine-grained control over extraction.

    ```bash
    vizb bench.txt --group-regex "Hash(?<n>[A-Za-z]+)(?<x>\d+)" -o output.html
    ```

    Best for custom naming conventions or when you need to skip fixed prefixes and suffixes.
  </TabItem>
</Tabs>

<Aside type="tip">
  Either method splits the label the same way â€” so for CSV/JSON, remember you still need `--group` to pick the columns first. Benchmarks need no `--group` to chart, since the name is already the label.
</Aside>
