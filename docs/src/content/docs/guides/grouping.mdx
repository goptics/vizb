## What is Grouping

Grouping turns each row into a point with up to four dimensions: **Name**, **XAxis**, **YAxis**, and **ZAxis**. This lets vizb chart the data meaningfully instead of as a flat list.

It always works in two stages:

1. **Build a label** for the row.
   - For **CSV/JSON**, you pick columns with `--group`. Vizb joins their per-row values using the same separators declared in `--group-pattern`.
   - For **benchmarks**, the benchmark name is the label already (e.g. `BenchmarkSort/1024/QuickSort`).
2. **Split the label** into dimensions with `--group-pattern` (separator-based) or `--group-regex` (named captures). This stage is identical for both inputs.

The two flows differ only in stage 1 — where the label comes from. The same pattern and regex syntax serves both.

## Why Grouping

Grouping creates multi-dimensional charts. Without it, every row is an isolated bar or point. With it, vizb can:

- **Separate charts** by logical category (e.g. one chart per algorithm or region).
- **Populate axes** with meaningful values (e.g. input size on the X-axis).
- **Create series** for variants (e.g. different data distributions on the Y-axis).

| Without Grouping | With Grouping |
|---|---|
| Single flat chart with all rows | Multiple charts grouped by name |
| No axis structure | X-axis and Y-axis with meaningful categories |
| Hard to compare across dimensions | Easy side-by-side comparison |

## The Dimensions

Grouping extracts up to four dimensions from each label:

| Dimension | Role | Description |
|---|---|---|
| **Name** (`name` / `n`) | Groups rows into separate charts | Rows sharing the same name appear in the same chart |
| **XAxis** (`xAxis` / `x`) | Sets the X-axis categories | Each unique value becomes a tick on the X-axis |
| **YAxis** (`yAxis` / `y`) | Creates series / variants | Each unique value becomes a separate series or bar group |
| **ZAxis** (`zAxis` / `z`) | Adds a depth axis → 3D chart | Each unique value becomes a stacked depth layer in a [3D chart](/charts/3d) |

<Aside type="note">
  At least one axis (`x` or `y`) is required for grouping to take effect. The `name` dimension is optional. When omitted, all rows go into a single chart. The `z` dimension requires **both** `x` and `y` to be present.
</Aside>

## Grouping CSV/JSON data

CSV and JSON rows have no benchmark name. You tell vizb which columns carry the structure. The `--group` / `-g` flag selects columns. The `--group-pattern` / `-p` flag assigns each column to a dimension. By default each column maps to one dimension (`-p y,x`). Use **bracket slots** `[...]` when a single column's **cell value** encodes multiple dimensions (dates, slash paths, underscore IDs, etc.).

### Auto-grouping (no flags needed)

If you run vizb on a CSV or JSON file **without** any of `--group`, `--group-pattern` (beyond its default), or `--group-regex`, vizb infers the category axis for you so `vizb data.csv` "just works":

1. It collects **non-numeric columns** as candidates (numeric columns are ignored entirely).
2. It ranks candidates by **distinct-value count** (highest cardinality wins; ties go to the leftmost column) and picks the top one as the **XAxis** (`-p x`).

The choice is logged to stdout (`🧠 Auto-grouped by column: date`), so you can override it with an explicit `--group` whenever the inference is wrong. Benchmark parsers (Go, Rust, JavaScript) are unaffected — auto-grouping applies only to the `csv` and `json` parsers.

Grouped 3D charts need both `x` and `y` in `--group-pattern`. Auto-grouping always sets `-p x` only, so `--3d` with auto-group logs a warning and grouped 3D will not render until you supply both axes explicitly (e.g. `-g region,product -p x,y --3d`).

```bash
vizb sales.csv -o sales.html                              # auto-picks region (or similar) as the X axis
vizb sales.csv --chart bar:3d -o 3d.html                  # auto-picks X only; warns that 3D needs -p x,y
vizb sales.csv -g region,product -p x,y --3d -o 3d.html   # grouped 3D with explicit axes
vizb sales.csv -g product -p y -o o.html                  # explicit --group disables auto-grouping
```

### Auto-value mode (all-numeric data)

When every column in a CSV/JSON file is numeric (no categorical columns exist), auto-grouping has nothing to pick. Instead, vizb **auto-assigns the first 2-3 columns** as value-type coordinate axes (`x`, `y`, `z`) — no flag needed. This works on **bar**, **line**, and **scatter** charts.

| Columns | Axes | Pattern |
|---|---|---|
| 2+ | `x`, `y` | 2D value |
| 3+ | `x`, `y`, `z` (first 3) | **3D value** (auto-enables 3D) |
| 4+ | `x`, `y`, `z` + 4th as **metric** | **3D value** + auto visualMap (color/size by metric) |

With 2+ numeric columns, vizb creates a 2D value-mode chart. With 3+, it automatically enables 3D (`bar3D` / `line3D` / `scatter3D`). With 4+, the first three columns are coordinates and the fourth becomes a visual metric. The inference is logged: `🧠 Auto-valued by columns: x, y, z (3D pattern x-y-z), metric: value`.

```bash
vizb scatter allnums.csv -o scatter.html        # auto-value x,y from first 2 columns
vizb scatter allnums.csv --3d -o 3d.html         # auto-value x,y,z from 3+ columns (auto-3D)
vizb bar allnums.csv -o bars.html                # bar auto-value (auto-3D with 3+ cols)
vizb line allnums.csv -o lines.html              # line auto-value (auto-3D with 3+ cols)
```

### Solo `--select` axis mode (no grouping)

When you pass `--select` **without** explicit grouping (`-g`, `-r`, or non-default `-p`), vizb enters **select axis mode**. Each repeatable `--select` defines one view with 2–3 columns assigned to `x`, `y`, and optional `z`. Both auto-group and auto-value are disabled.

| Selected columns | Result |
|---|---|
| All numeric | **Value mode** — continuous coordinate axes |
| One categorical + numeric | **Mixed mode** — category on x, values on y[,z]; [scatter](/charts/scatter) is the primary chart |
| Repeatable flags (2 cols each) | One dataset; each flag → stat type (`latency by region`); chart tabs by stat |

```bash
# Value mode: pick two numeric columns as x,y
vizb scatter spiral-3d.csv --select x,y -o 2d.html

# Mixed mode: category region + numeric latency
vizb scatter examples/csv/region-metrics.csv --select region,latency -o mixed.html

# Multi-stat: one dataset, charts by stat type
vizb scatter examples/csv/region-metrics.csv \
  --select region,latency \
  --select region,sales \
  -o dual.html
```

Use explicit syntax when you need to pin the category column: `--select x:region,y:latency,z:sales`.

### Choosing: `--group` + select vs solo `--select`

Use this decision guide when your CSV/JSON has both categorical and numeric columns:

| Goal | Approach | Example |
|---|---|---|
| Compare metrics **across categories** with legend series | Grouped + numeric `--select` | `-g region,product -p x,y --select latency,sales` |
| Plot **one metric per category label** on a coordinate chart | Solo mixed `--select` | `--select region,latency` → scatter |
| Plot **raw numeric coordinates** (no categories) | Solo value `--select` or auto-value | `--select x,y` or no flags on all-numeric file |
| **Several metric views** from one file (no grouping) | Repeatable solo `--select` | `--select region,latency --select region,sales` → charts `latency by region`, `sales by region` |
| Split a column into year/month/day on axes | Grouped with bracket `-p` | `-g date,category -p '[n-y-x],z'` |
| Quick chart with no flags on mixed data | Auto-group | `vizb bar sales.csv` picks highest-cardinality categorical column |

<Aside type="tip">
  **Grouped** mode aggregates rows that share the same `(name, x, y, z)` key. **Solo `--select`** keeps every row as a separate point — no aggregation. Choose grouped when your table has many rows per dimension combination; choose solo select when each row is already one observation.
</Aside>

### Matching separators

For tabular data, `-g` and `-p` must use the same separators between column slots. If they disagree, vizb exits before parsing:

```text
--group "<g>" and --group-pattern "<p>" separators do not match (expected "<from g>", got "<from p>")
```

The `expected` side is derived from how `-g` declares column boundaries. The `got` side is what `-p` declares. Fix whichever flag is wrong so both use the same separator sequence.

| Mismatch | Error (abbreviated) |
|---|---|
| `-g region,product,category -p x/y/z` | `expected ", ,", got "/ /"` — comma `-g` needs comma `-p` (`x,y,z`) |
| `-g region/product/category -p x,y/z` | `expected "/ /", got ", /"` — slash `-g` needs slash `-p` (`x/y/z`) |
| `-g "region#product#category" -p x#y,z` | `expected "# #", got "# ,"` — every separator in `-g` must appear in `-p` at the same positions |

| `--group` style | Matching `-p` | Row label (example) |
|---|---|---|
| Comma list (`-g region,product`) | Commas (`-p y,x`) | `Asia,Widget` |
| Quoted string (`-g "region product"`) | Spaces (`-p "y x"`) | `UAE Gadget` |
| Single value (`-g "region/product/category"`) | Same separators (`-p x/y/z`) | `EU/Widget/Sports` |
| Structured (`-g region,product/category`) | Same shape (`-p y,x/z`) | `Asia,Widget/Electronics` |

Comma-separated `-g` (the usual pflag form) **requires** comma-separated `-p`. For example, use `-g region,product,category` with `-p x,y,z`, not `x/y/z`.

When `-g` and `-p` define a different **number** of slots (common with bracket patterns), vizb reports a separate dimension-count error:

```text
--group and --group-pattern dimension count do not match: N column(s) in "<g>", M slot(s) in --group-pattern "<p>"
```

Take this `sales.csv`:

| region | product | category | sales | latency |
|---|---|---|---|---|
| Asia | Widget | Electronics | 100 | 12 |
| UAE | Gadget | Home | 80 | 15 |
| EU | Gizmo | Sports | 60 | 11 |

```bash
vizb sales.csv --group region,product -p y,x -o sales.html
```

For the first row, vizb joins the selected columns into the label `Asia,Widget`. Then the pattern `y,x` splits it:

| Selected columns | Label | Pattern `y,x` | Result |
|---|---|---|---|
| `region`, `product` | `Asia,Widget` | `y` → `Asia`, `x` → `Widget` | YAxis = Asia, XAxis = Widget |

**Space-separated columns** — pass one quoted `-g` value and use spaces in `-p`:

```bash
vizb sales.csv -g "region product" -p "y x" -o sales.html
```

**Structured grouping** — mix separators inside `-g` (commas between pflag values, `/` or `_` inside a value) to declare a hierarchy. `-p` must mirror the same shape:

```bash
vizb sales.csv -g region,product/category -p y,x/z -o sales.html
vizb sales.csv -g "region/product/category" -p x/y/z -o sales.html
```

### Bracket slots — split a column's cell value

When one column holds an encoded string (not separate columns), wrap the inner pattern in `[...]`. Each **top-level** slot lines up 1:1 with a `-g` column:

| Slot form | Meaning |
|---|---|
| `x`, `y`, `n`, `z` | Whole cell value → that dimension |
| `x{label}` | Whole cell → dimension, axis titled **label** (overrides column name) |
| `[inner]` | Split the cell value with **inner** separators → multiple dimensions |
| `n{year}-y{months}` inside `[...]` | Split value **and** set per-dimension axis titles |

Top-level separators between slots still match `-g` (commas for `-g a,b`, spaces for `-g "a b"`). **Benchmarks do not use brackets**. Keep flat `-p n/x/y` for `bench.txt`.

<Aside type="tip">
  Use **single quotes** around `-p` when it contains `[...]` bracket slots:

  ```bash
  vizb sales.csv -g date,category -p '[n{year}-y{months}-x{dates}],z' -o sales.html
  ```
</Aside>

Bracket slots can attach `{label}` to each inner dimension — see [Axis labels](#axis-labels).

**Date + category** — split `2022-2-30` into year/month/day, keep category whole:

```bash
vizb sales.csv -g date,category -p "[x-y-n],z" -o sales.html
```

| date | category | Pattern | Result |
|---|---|---|---|
| `2022-2-30` | `Widget` | `[x-y-n]` splits on `-`, `z` ← category | x=2022, y=2, n=30, z=Widget |

<details>

**ISO dates (`YYYY-MM-DD`)** — use a bracket slot when the date is one column. `{year}`, `{months}`, and `{dates}` only set axis titles. Splitting is driven by `-` between dimension tokens.

Full year / month / day:

```bash
vizb sales.csv -g date,category -p '[n{year}-y{months}-x{dates}],z' -o sales.html
```

| date | Pattern | Result |
|---|---|---|
| `2024-01-01` | `[n{year}-y{months}-x{dates}]` | n=2024, y=01, x=01 |

Drop the **last** segment (day) — add a **trailing** separator after the last token:

```bash
vizb sales.csv -g date -p '[n{year}-x{months}-]' -o sales.html
```

| date | Pattern | Result |
|---|---|---|
| `2024-01-01` | `[n{year}-x{months}-]` | n=2024, x=01 (day dropped) |

Skip the **middle** segment (month) — use **consecutive** separators (`--`) between tokens:

```bash
vizb sales.csv -g date -p '[n{year}--x{dates}]' -o sales.html
```

| date | Pattern | Result |
|---|---|---|
| `2024-02-15` | `[n{year}--x{dates}]` | n=2024, x=15 (month skipped) |

On `2024-01-01` the month and day are both `01`. So year+month and year+day patterns look the same. Use a date like `2024-02-15` to confirm which segment maps to `x`.

**Slash-encoded benchmark column** — same layout as Go bench names, stored in CSV:

|---|---|---|
| `Sort/1024/QuickSort` | `[n/x/y]` | n=Sort, x=1024, y=QuickSort |

| id | Result |

| `lib_func_case` | n=lib, y=func, x=case |

**Dot-separated version on X/Y:**

```bash
vizb releases.csv -g version -p "[x.y]" -o out.html
```

| version | Result |
|---|---|
| `1.2.3` | x=1, y=2 (remaining segment unused) |

**Space-separated label in one column:**

```bash
vizb data.csv -g "label region" -p "[x n y],z" -o out.html
```

| label | region | Result |
|---|---|---|
| `alpha beta gamma` | `EU` | x=alpha, n=beta, y=gamma, z=EU |

| name | Result |
|---|---|
| `BenchmarkTest/JSON/Marshal` | n=JSON, y=Marshal |

|---|---|---|

| tuple | Result |
|---|---|

<Aside type="note">
  Bracket splitting is **literal**. Vizb does not parse dates or semver semantically. Values must match the separator layout in `-p`. For irregular strings without stable separators (e.g. `Hashing64MD5`), use [`--group-regex`](#regex-syntax) instead.
</Aside>

The columns you **don't** select decide the chart values:

- **Unselected numeric columns become series.** Here `sales` and `latency` each become a series (one `Stat` per column).
- **Non-numeric unselected columns are ignored.** A `notes` text column would simply be dropped.
- **No `--group` at all** → dimensions stay empty. No rows are merged. Each row becomes one point carrying every numeric column as a series. This is useful for a quick chart of a wide table.

<Aside type="note">
  Each `--group` column must match a header exactly. A missing column is fatal: `group column 'X' not found; available: [...]`.
</Aside>

### Aggregation

When `--group` is set, multiple rows can share the same `(name, x, y, z)` key. Vizb **sums** their values into a single point. A 200k-row export collapses into a few thousand chart points. You'll see it in the CLI output:

```text
🧮 Aggregating 50000 rows by columns: region, product, month (name: region, x: product, y: month)...
✅ Aggregated into 1200 grouped data points
```

<Aside type="caution">
  Aggregation is **summing**, and applies to **CSV/JSON only**. Benchmark parsers are never aggregated. Repeated runs share a key but the UI averages them instead of summing.
</Aside>

See [Tabular Data (CSV & JSON)](/guides/data) for the full parser rules.

## Grouping benchmarks

Benchmark names already encode structure. A name like `BenchmarkSort/1024/QuickSort` carries the function, the input size, and the variant. There are no columns to pick. So `--group-pattern` / `--group-regex` do all the extraction. Set axis titles with `{label}` suffixes in `-p` (see [Axis labels](#axis-labels)).

```bash
go test -bench . | vizb -p n/x/y -o output.html
```

| Benchmark Name | Name | XAxis | YAxis |
|---|---|---|---|
| `BenchmarkSort/1024/QuickSort` | Sort | 1024 | QuickSort |
| `BenchmarkSort/2048/MergeSort` | Sort | 2048 | MergeSort |

Every metric the benchmark reports (time, memory, allocations, …) becomes a series automatically. You only group the name.

## Pattern Syntax

The `--group-pattern` flag splits the label using the separators you declare between dimension tokens. It assigns each segment to a dimension. It applies to both flows — the benchmark name or the CSV/JSON label built from `--group`.

### Separators

Any single character works between dimension tokens. Vizb reads them from your `-p` string:

| Separator | Typical use |
|---|---|
| `/` | Go/Rust/JS benchmark names (`n/x/y`) and slash-structured `-g` (`region/product`) |
| `_` | Underscore-delimited bench names (`n_y_x`) and structured `-g` specs |
| `,` | Comma-separated `--group` lists (`y,x` with `-g region,product`) |
| ` ` (space) | Quoted `--group` specs (`"y x"` with `-g "region product"`) |
| `#`, `.`, etc. | Any single character works when `-g` and `-p` agree (e.g. `-g "a#b#c" -p "x#y#z"`) |

Three skip patterns drop segments without assigning them to a dimension:

| Pattern | Example | Effect |
|---|---|---|
| Leading separator | `/n/y` | First value segment is unused (prefix skip) |
| Consecutive separators | `n--x` or `x//y` | Middle segment is unused |
| Trailing separator | `n-x-` | Last value segment is unused |

Each repeated separator character adds one more split point and one empty dimension slot. A single trailing `-` after the last token is enough to drop the final segment. You do not need `---`.

Works in flat benchmark patterns and inside bracket slots (e.g. `[n{year}-x{month}-]` on a CSV date column).

### Components

| Component | Shorthand | Meaning |
|---|---|---|
| `name` | `n` | Chart group name |
| `xAxis` | `x` | X-axis values |
| `yAxis` | `y` | Y-axis series |
| `zAxis` | `z` | Z-axis depth layer (3D, requires `x` + `y`) |

The default pattern is `x` — a single segment mapped to the X-axis.

### Examples

**Slash-separated `n/x/y`:**

```bash
vizb bench.txt --group-pattern "n/x/y" -o output.html
```

| Label | Name | XAxis | YAxis |
|---|---|---|---|
| `BenchmarkSort/1024/QuickSort` | Sort | 1024 | QuickSort |
| `BenchmarkSort/2048/MergeSort` | Sort | 2048 | MergeSort |
| `BenchmarkSort/4096/QuickSort` | Sort | 4096 | QuickSort |

**Underscore-separated `n_y_x`:**

```bash
vizb bench.txt --group-pattern "n_y_x" -o output.html
```

| Label | Name | YAxis | XAxis |
|---|---|---|---|
| `BenchmarkHash_SHA256_1KB` | Hash | SHA256 | 1KB |
| `BenchmarkHash_MD5_4KB` | Hash | MD5 | 4KB |

**2D with `n/y` (no X-axis grouping):**

```bash
vizb bench.txt --group-pattern "n/y" -o output.html
```

| Label | Name | YAxis |
|---|---|---|
| `BenchmarkJSON/Marshal` | JSON | Marshal |
| `BenchmarkJSON/Unmarshal` | JSON | Unmarshal |

**Prefix skip with `/n/y` (leading slash discards the first segment):**

```bash
vizb bench.txt --group-pattern "/n/y" -o output.html
```

| Label | Name | YAxis |
|---|---|---|
| `BenchmarkTest/JSON/Marshal` | JSON | Marshal |
| `BenchmarkTest/XML/Parse` | XML | Parse |

**Middle skip with `n//y` (consecutive separators discard the middle segment):**

```bash
vizb bench.txt --group-pattern "n//y" -o output.html
```

| Label | Name | YAxis |
|---|---|---|
| `Sort/1024/QuickSort` | Sort | QuickSort |

**Trailing skip with `n-x-` (trailing separator discards the last segment):**

```bash
vizb bench.txt --group-pattern "n-x-" -o output.html
```

| Label | Name | XAxis |
|---|---|---|
| `2024-01-01` | 2024 | 01 |

**4D with `n/x/y/z` (renders a 3D chart):**

```bash
vizb bench.txt --group-pattern "n/x/y/z" -o output.html
```

| Label | Name | XAxis | YAxis | ZAxis |
|---|---|---|---|---|
| `BenchmarkSort/1024/QuickSort/warm` | Sort | 1024 | QuickSort | warm |
| `BenchmarkSort/2048/MergeSort/cold` | Sort | 2048 | MergeSort | cold |

<Aside type="note">
  A pattern must contain `x` or `y` (`pattern must contain xAxis (x) or yAxis (y)`). A `z` segment requires both `x` and `y`. Vizb rejects `z` otherwise with `zAxis (z) requires both xAxis (x) and yAxis (y)`. See [3D Charts](/charts/3d).
</Aside>

## Regex Syntax

The `--group-regex` flag uses Go-style named capture groups. It runs on the label — a benchmark name, or the CSV/JSON label built from `--group`. Use it when the label does not follow a simple separator pattern. When set, regex takes precedence over `--group-pattern`.

### Named Capture Groups

| Group Name | Shorthand | Dimension |
|---|---|---|
| `(?<name>...)` | `(?<n>...)` | Chart group name |
| `(?<xAxis>...)` | `(?<x>...)` | X-axis values |
| `(?<yAxis>...)` | `(?<y>...)` | Y-axis series |
| `(?<zAxis>...)` | `(?<z>...)` | Z-axis depth layer (3D, requires `x` + `y`) |

### Examples

| Label | Regex | Extracted |
|---|---|---|
| `BenchmarkHashing64MD5` | `Hashing64(?<x>.*)` | X: MD5 |
| `BenchmarkJSONByMarshal` | `(?<x>.*)By(?<y>.*)` | X: JSON, Y: Marshal |
| `BenchmarkDecode/text=digits/level=speed` | `(?<n>.*)/text=(?<x>.*)/level=(?<y>.*)` | N: Decode, X: digits, Y: speed |

```bash
vizb bench.txt --group-regex "(?<n>.*)/text=(?<x>.*)/level=(?<y>.*)" -o output.html
```

<Aside type="note">
  The regex must capture `x` or `y` (`regex '...' does not contain x (xAxis) or y (yAxis)`). A captured `z` still requires both `x` and `y`.
</Aside>

## Axis labels

Set chart axis titles in `--group-pattern` / `-p` by appending `{title}` immediately after a dimension token (no space):

| Shorthand | Syntax | Chart title |
|---|---|---|
| `n` | `n{Algorithm}` | Algorithm |
| `x` | `x{Product}` | Product |
| `y` | `y{Region}` | Region |
| `z` | `z{Category}` | Category |

Works on flat patterns, bracket slots, and benchmarks. The same syntax is used everywhere `-p` is used.

```bash
# CSV — titles independent of column names
vizb sales.csv -g region,product -p 'y{Region},x{Product}' -o sales.html

# Bracket slot — one column, multiple titled dimensions
vizb sales.csv -g date,category -p '[n{year}-y{months}-x{dates}],z{Category}' -o sales.html

# Benchmark — no --group required
vizb bench.txt -p 'n{Algorithm}/x{Size}/y{Variant}' -o out.html
```

| `-p` fragment | Default title (no `{}`) | With `{label}` |
|---|---|---|
| `y` with `-g region,product` | `region` (column name) | `y{Region}` → **Region** |
| `x` with `-g region,product` | `product` (column name) | `x{Product}` → **Product** |
| `[n{year}-y{months}-x{dates}]` | `date` for each split dim | **year**, **months**, **dates** |
| `n/x/y` on benchmarks | (empty) | `n{Algorithm}/x{Size}/y{Variant}` → titled axes |

Omit `{label}` on CSV/JSON and the matching `--group` column name becomes the axis title. When `{label}` is present, it always overrides that default.

<Aside type="note">
  Titles are emitted as `axisLabels` in the JSON output and rendered on chart axes. The `name` title is preserved for the UI axis-swap feature.
</Aside>

## Choosing a Method

<Tabs syncKey="method">
  <TabItem label="Pattern" icon="list">
    Use `--group-pattern` when the label follows a consistent separator format.

    ```bash
    # Benchmarks — slash-separated names
    vizb bench.txt --group-pattern "n/x/y" -o output.html

    # CSV/JSON — comma-separated columns
    vizb sales.csv -g region,product -p y,x -o sales.html
    ```

    Best for standard Go benchmarks (`BenchmarkFunc/size/variant`) and for CSV/JSON where columns map cleanly to dimensions with matching separators.
  </TabItem>
  <TabItem label="Regex" icon="search">
    Use `--group-regex` when the label has a non-standard format or you need fine-grained control over extraction.

    ```bash
    vizb bench.txt --group-regex "Hash(?<n>[A-Za-z]+)(?<x>\d+)" -o output.html
    ```

    Best for custom naming conventions or when you need to skip fixed prefixes and suffixes.
  </TabItem>
</Tabs>

<Aside type="tip">
  Either method splits the label the same way. For CSV/JSON, remember you still need `--group` to pick the columns first. Benchmarks need no `--group` to chart, since the name is already the label.
</Aside>
